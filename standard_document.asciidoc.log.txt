[big]*i.     Abstract*

This OGC Standard defines EmissionML, a conceptual data model for representing atmospheric emission events. EmissionML provides a domain-specific ontology for describing the release of substances into the atmosphere, including the source from which an emission originates, the mechanism by which it occurs, the quantity of substance released, and the temporal bounds of the event.

EmissionML distinguishes itself through three key design principles. First, it incorporates an evidence chain linking emission determinations to supporting Observations, enabling traceability from reported values to underlying measurements. Second, it provides explicit uncertainty provenance by attaching data quality metadata to both emission quantities and temporal bounds, ensuring that the confidence and precision of each determination are recorded alongside the values themselves. Third, it offers clear spatial and temporal context — spatial through Feature (as the ultimate feature of interest) and Source (as the emission origin), temporal through start time, end time, and the phenomenon time of supporting Observations. By adopting the OMS 3.0 Observation model, EmissionML provides a common framework for reconciling heterogeneous sensor observations — from remote sensing satellites to ground-based continuous monitoring stations — within a single, unified evidence chain.

The model builds on foundational OGC and ISO standards. Source inherits from Feature (ISO 19109), DeterminationProcedure inherits from Procedure (ISO/OGC 19156:2023), and Observations conform to the OMS 3.0 Observation model (ISO/OGC 19156:2023). The standard defines twelve requirements specifying the structure and constraints of the core data model.

By making emission semantics, evidence chains, and uncertainty provenance explicit and machine-verifiable, EmissionML enables credibility of reported emission values, traceability from determinations to their supporting observations and procedures, and reproducibility of emission assessments across organizations and jurisdictions.

The intended audience includes environmental regulators, greenhouse gas reporting frameworks, emissions monitoring system developers, and geospatial data infrastructure implementers who require a standardized, interoperable representation of emission event data.

== Terms and Definitions
This document uses the terms defined in link:++https://portal.ogc.org/public_ogc/directives/directives.php++[OGC Policy Directive 49], which is based on the ISO/IEC Directives, Part 2, Rules for the structure and drafting of International Standards. In particular, the word “shall” (not “must”) is the verb form used to indicate a requirement to be strictly followed to conform to this Standard and OGC documents do not use the equivalent phrases in the ISO/IEC Directives, Part 2.

This document also uses terms defined in the OGC Standard for Modular specifications (link:++https://portal.opengeospatial.org/files/?artifact_id=34762++[OGC 08-131r3]), also known as the 'ModSpec'. The definitions of terms such as standard, specification, requirement, and conformance test are provided in the ModSpec.

For the purposes of this document, the following additional terms and definitions apply.

Determination Procedure::
A Determination Procedure describes the steps performed to determine the quantity or temporal bounds of an Emission Event.

Emission Event::
An Emission Event represents the release of a substance into the atmosphere over a non-zero duration. It is characterized by its start time and end time, the Source Feature from which the substance originated, and the identity and quantity of the emitted substance.

Emission Quantity::
An Emission Quantity is the physical quantity representing the total mass of substance released to the atmosphere during a defined Emission Event or over a specified time interval.

Emission Intent::
Emission Intent is a code list describing whether an emission is intentional, unintentional, or unknown.

Feature::
Abstraction of real world phenomena (Source: ISO 19101-1:2014)

Measurement::
Set of operations having the object of determining the value of a quantity (Source: link:++https://docs.ogc.org/as/20-082r4/20-082r4.html++[ISO/OGC 19156:2023])

Mechanism::
A Mechanism is a classification of the specific physical, chemical, or operational process that constitutes the direct cause or pathway for an emission. It is defined independently of the Source—including its identity, structure, or purpose—allowing Mechanisms to be consistently reused across different types of Source.

Observation::
Act carried out by an observer to determine the value of an observable property of an object (feature-of-interest) by using a procedure, with the value provided as the result (Source: link:++https://docs.ogc.org/as/20-082r4/20-082r4.html++[ISO/OGC 19156:2023])

Procedure::
Description of steps performed (Source: link:++https://docs.ogc.org/as/20-082r4/20-082r4.html++[ISO/OGC 19156:2023])

Source::
A Source represents the feature from which an emission originates. A Source is an ISO 191xx feature whose geometry provides the geospatial reference for one or more Emission Events. A Source is characterised by its own identity, geometry, intended function, and lifecycle, and is explicitly agnostic to the Mechanism by which any associated Emission Event occurs.

Source Type::
A Source Type classifies a Source by its category.

Temporal Bound::
A Temporal Bound is a value object representing a temporal determination (e.g., start time or end time) that MAY have associated data quality metadata.

Unit of Measure::
Reference quantity chosen from a unit equivalence group (Source: link:++https://docs.ogc.org/as/20-082r4/20-082r4.html++[ISO/OGC 19156:2023])

=== Emission Event Class

An Emission Event represents the release of a substance into the atmosphere over a non-zero duration. It is characterized by its start time and end time, the Source Feature from which the substance originated, and the identity and quantity of the emitted substance.

NOTE: Conceptually, an Emission Event corresponds to an occurrent in upper-ontology terms, as it unfolds over time and has temporal parts.

NOTE: For reporting or accounting purposes, a long-running emission event is often segmented into multiple time-bounded emission events (e.g., annual records). This segmentation reflects reporting conventions and does not imply physical cessation or re-initiation of the emission.

=== Source Class

A Source represents the feature from which an emission originates. A Source is an ISO 191xx feature whose geometry provides the geospatial reference for one or more Emission Events. A Source is characterised by its own identity, geometry, intended function, and lifecycle, and is explicitly agnostic to the Mechanism by which any associated Emission Event occurs.

NOTE: A Source is a continuant that may exist independently of any emission, whereas an Emission Event is an occurrent. The identity and lifecycle of a Source are independent of the existence, absence, or frequency of Emission Events.

=== Observation Class

An Observation is an act carried out by an observer to determine the value of an observable property of an object (feature-of-interest) by using a procedure, with the value provided as the result (Source: ISO/OGC 19156:2023). In EmissionML, an Observation is evidence that may support an Emission Quantity or Temporal Bound determination.

NOTE: The Ultimate Feature of Interest of an Observation identifies a Feature — which may be a Source or another feature at a different spatial scale — associated with the Emission Event. It is common for a single Emission Event to be observed by multiple Observations whose Ultimate Features of Interest are defined at different spatial scales. For example, a satellite remote sensing image Observation typically has an area-based Ultimate Feature of Interest (e.g., a facility area or region), whereas the same Emission Event may also be observed by a person carrying an Optical Gas Imaging camera, for which the Ultimate Feature of Interest is a specific Source such as a component of an item of equipment from which pollutants are released.

== Relationship with OGC Features and Observations

EmissionML is a domain-specific conceptual model that builds on two foundational OGC concepts: *Features* (ISO 19109 General Feature Model) and *Observations* (ISO/OGC 19156:2023, OMS 3.0). This annex explains how EmissionML relates to each and why explicit domain modeling is preferred over generic usage.

<<fig-annex-b-layering>> illustrates the conceptual layering of EmissionML on top of these foundational standards.

[[fig-annex-b-layering]]
.Conceptual layering of EmissionML on foundational ISO/OGC standards
image::images/annex-b-layering.png[]

=== EmissionML and OGC Features

The OGC General Feature Model (ISO 19109) defines a _Feature_ as an abstraction of a real-world phenomenon — any entity with an identity, properties, and optionally a geometry. EmissionML makes this alignment explicit by introducing a concrete *Feature* class (stereotyped `<<ISO 19109>>`) into the conceptual model.

The Feature class carries a mandatory `geometry` attribute of type `GM_Object` and serves as the base class for domain-specific feature types. *Source* inherits from Feature, gaining its geometry by inheritance. This structural inheritance — rather than a notional stereotype — makes the ISO 19109 alignment unambiguous and machine-verifiable.

Because Feature is concrete, an Observation's `ultimateFeatureOfInterest` can reference any Feature — not only a Source. This enables observations whose feature of interest is defined at a different spatial scale (e.g., a facility area, a region, or a monitoring zone) to participate in the evidence chain without being forced into the Source class.

*EmissionEvent* is also a real-world phenomenon — an occurrent that unfolds over time. In principle, an Emission Event _could_ be modeled as a generic Feature. However, EmissionML gives it explicit structure: temporal bounds (start time, end time), a quantity of emitted substance, a mechanism describing the physical or operational process, an intent classification, and an evidence chain linking the event to supporting Observations.

Without this explicit modeling, a Source would be indistinguishable from any other Feature, and an emission event would be "just another Feature" with ad-hoc properties. The semantic relationships that give emission data its meaning — `hasSource`, `hasMechanism`, `hasQuantity`, `hasStartTimeEvidence` — would be lost or require convention-based interpretation.

=== EmissionML and OMS 3.0 Observations

OMS 3.0 (ISO/OGC 19156:2023) defines an Observation as an act of determining the value of an observable property. An Observation has a well-defined structure: a feature of interest, an observed property, a procedure, and a result.

EmissionML incorporates OMS 3.0 Observation *by reference* (Requirement 8) — it does not redefine or extend the Observation class. In EmissionML, Observations serve as *evidence* supporting emission quantity and temporal bound determinations, linked through the `hasQuantityEvidence`, `hasStartTimeEvidence`, and `hasEndTimeEvidence` associations.

One might ask: _Why not model the entire emission event as an Observation result?_ Because an Emission Event is richer than what a single Observation captures. An Emission Event has a source, a mechanism, an intent, multiple temporal bounds, and can be supported by _multiple_ Observations at different spatial and temporal scales. For example:

* A satellite remote sensing Observation may detect a methane plume over a broad area.
* An Optical Gas Imaging (OGI) camera Observation may pinpoint the leak to a specific component.
* A continuous emissions monitoring system (CEMS) Observation may quantify the emission rate over time.

All three Observations can serve as evidence for a single Emission Event, each contributing to the determination of quantity, start time, or end time. The Observation's `ultimateFeatureOfInterest` references a Feature — which may be a Source or any other Feature at a different spatial scale. The satellite Observation's feature of interest may be a Feature representing an entire facility, while the OGI Observation's feature of interest is a Source representing a single valve.

=== EmissionML and OMS 3.0 Procedures

OMS 3.0 defines a *Procedure* as "a description of steps performed" and introduces *ObservingProcedure* as a specialization that describes steps performed to determine the value of an observable property. EmissionML follows this same specialization pattern.

The EmissionML conceptual model introduces a *Procedure* class (stereotyped `<<ISO 19156:2023>>`) as the base class for process descriptions. *DeterminationProcedure* inherits from Procedure and describes the steps performed to determine a value — such as an emission quantity or temporal bound. This structural inheritance mirrors how ObservingProcedure specializes Procedure in OMS 3.0.

By making Procedure an explicit base class rather than a notional reference, EmissionML ensures that its process descriptions are structurally aligned with OMS 3.0 — not merely by naming convention, but by class hierarchy. This parallels the Feature/Source pattern: just as Source inherits from Feature to make ISO 19109 alignment concrete, DeterminationProcedure inherits from Procedure to make OMS 3.0 alignment concrete.

=== Why Explicit Domain Modeling

*The alternative.* One could model everything with generic Features and Observations — Sources as plain Features, emission data as Observation results, and relationships as ad-hoc properties. This approach uses only existing OGC standards without defining new classes.

*The problem.* Generic modeling loses domain semantics. Without EmissionML's explicit classes, systems cannot:

* distinguish a Source from any other Feature,
* enforce that an emission has a Mechanism and an Intent,
* trace the evidence chain from Observation to Emission Quantity to Emission Event,
* validate that required relationships are present (e.g., every Emission Event must have a Source, a Mechanism, and a Determination Procedure).

*What explicit modeling provides:*

Interoperability:: Systems exchanging EmissionML data share a common vocabulary. EmissionEvent, Source, Mechanism, and EmissionQuantity are unambiguous — they mean the same thing in every implementation.

Validation:: Requirements can be formally specified and enforced. For example, Requirement 1 mandates that every Emission Event has a Source, a Mechanism, start time, quantity, and their corresponding Determination Procedures.

Usability:: Domain experts work with emission concepts directly — _emission events_, _sources_, _mechanisms_ — rather than mapping to and from generic abstractions. This reduces the cognitive burden and the risk of misinterpretation.

*Analogy.* Just as CityGML defines explicit Building, Road, and Vegetation classes rather than using generic Features, EmissionML defines explicit emission domain classes. The General Feature Model provides the abstract foundation; EmissionML provides the domain-specific vocabulary that makes emission data interoperable and actionable.
